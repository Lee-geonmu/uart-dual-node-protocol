# UART Dual-Node Protocol

Arduino Nano ↔ Uno 2노드 사이에 자체 설계한 프레임 프로토콜(CRC-8, 시퀀스 번호, ACK/NACK 재전송)을 얹어
손상·유실·중복까지 대응하는 신뢰성 UART 통신을 구현한 프로젝트.

## 시연 영상

▶ [시연 영상 보기]

정상 통신 → 강제 단선(재전송·실패 판정) → 재연결 복구 → ACK 유실 시 중복 프레임 처리까지 실제 동작을 담았다.

## 왜 만들었나

SoftwareSerial로 단순히 write/read만 하면 통신 자체는 되지만, 프레임이 깨지거나 사라져도 아무도 모른다.
실제 제품 레벨의 시리얼 통신은 손상 검출, 유실 복구, 중복 방지가 기본으로 들어간다.
STM32 하드웨어 UART와 CAN으로 넘어가기 전에, 이 세 가지를 소프트웨어 레벨에서 직접 설계해 보는 것이 목적이다.

## 하드웨어

| 부품 | 수량 |
|---|---|
| Arduino Uno | 1 |
| Arduino Nano (CH340 호환) | 1 |
| 가변저항 10kΩ | 1 |
| LED (초록/빨강) | 4 |
| 저항 220Ω | 4 |
| 브레드보드, 점퍼선 | - |

배선:

| Nano | 방향 | Uno |
|---|---|---|
| D3 (TX) | → | D8 (RX) |
| D2 (RX) | ← | D9 (TX) |
| GND | — | GND |

- 가변저항: 양끝 5V / GND, 와이퍼 → Nano A0
- LED: 양쪽 보드 공통으로 D6(초록, 220Ω) = 통신 정상, D5(빨강, 220Ω) = 통신 실패
- 하드웨어 UART(D0/D1)는 USB 시리얼 디버깅 전용으로 남기고, 노드 간 링크는 SoftwareSerial 9600bps 사용

## 프로토콜 설계

```
[START 0xAA][LEN][TYPE][SEQ][PAYLOAD ...][CRC-8][END 0x55]
```

| 필드 | 크기 | 설명 |
|---|---|---|
| LEN | 1B | TYPE + SEQ + PAYLOAD 길이 |
| TYPE | 1B | 0x01 DATA / 0x02 ACK / 0x03 NACK |
| SEQ | 1B | 프레임 번호. 재전송 시 동일 값 유지 |
| PAYLOAD | 0~N | DATA 프레임: 아날로그 값 2B (리틀엔디안) |
| CRC-8 | 1B | poly 0x07, LEN~PAYLOAD 구간 대상 |

동작 흐름:

1. Nano가 200ms마다 가변저항 값을 DATA 프레임으로 전송
2. Uno는 CRC 검증 후 통과 시 ACK, 실패 시 NACK 회신
3. Nano는 300ms 안에 ACK를 못 받으면 같은 SEQ로 재전송 (최대 3회)
4. 3회 초과 시 실패 판정 후 빨간 LED 점등, 다음 주기에는 새 SEQ로 최신 값 전송
5. Uno는 직전에 처리한 SEQ와 같은 프레임이 오면 ACK 유실로 인한 재전송으로 판단하고, 값은 반영하지 않은 채 ACK만 재전송
6. Uno는 2초 이상 유효 프레임이 없으면 통신 두절 표시

## 실험 1 — baud rate별 프레임 실패율

측정 방법: 전송 간격 50ms, ACK 타임아웃 100ms로 부하를 높인 스트레스 조건에서 1000프레임씩 전송, ACK 미수신을 실패로 카운트했다.

1차 측정 (baud 변경 시마다 재업로드, 각 3회):

| baud | 측정값 (%) | 평균 | 표준편차 |
|---|---|---|---|
| 9600 | 9.50 / 0.30 / 0.30 | 3.37 | 5.31 |
| 19200 | 1.90 / 1.40 / 0.70 | 1.33 | 0.60 |
| 38400 | 4.30 / 0.70 / 0.50 | 1.83 | 2.14 |

세 조건 모두 재업로드 직후 첫 실행의 실패율이 유독 높게 나오는 공통 패턴이 보였다.
baud rate 자체보다 이 요인이 결과를 흔들고 있다고 판단해, 변수를 통제한 재실험을 진행했다.

재실험 (최초 1회 업로드 후 매회 리셋 버튼만으로 5회 반복):

| baud | 측정값 (%) | 평균 | 표준편차 |
|---|---|---|---|
| 9600 | 0.00 / 0.20 / 0.60 / 0.30 / 0.60 | **0.34** | 0.26 |
| 19200 | 0.30 / 0.40 / 0.40 / 0.50 / 1.30 | 0.58 | 0.41 |
| 38400 | 0.20 / 0.30 / 0.80 / 0.30 / 0.40 | 0.40 | 0.23 |

결론:

- 재업로드 직후의 초기 불안정 요인을 통제하면 세 조건 모두 실패율 1% 미만
- 그중 9600이 평균 기준 가장 낮아 최종 링크 속도로 채택
- 1차 측정에서 보인 큰 편차의 정확한 원인은 규명하지 못했다. 다만 측정 조건이 결과에 미치는 영향이 baud rate 간 차이보다 컸다는 점 자체가 이번 실험의 주요 관찰이다

## 실험 2 — XOR 체크섬 vs CRC-8 오류 검출률

설계 초기에는 XOR 체크섬을 썼다가 CRC-8로 교체했고, 그 근거를 시뮬레이션으로 확인했다.

방법: 실제 DATA 프레임과 같은 5바이트 데이터에 1~4비트를 무작위로 반전시키고,
XOR / CRC-8 각각이 오류를 놓치는 비율(miss rate)을 측정. 케이스당 100,000회 반복.

| 반전 비트 수 | XOR miss (%) | CRC-8 miss (%) |
|---|---|---|
| 1 | 0.00 | 0.00 |
| 2 | **10.26** | 0.00 |
| 3 | 0.00 | 0.00 |
| 4 | 3.01 | 0.82 |

결론:

- XOR은 짝수 개 비트가 동시에 반전되는 경우를 구조적으로 놓친다 (2비트 오류의 10% 이상 미검출)
- CRC-8은 1~3비트 오류를 전부 잡았고 4비트에서도 miss가 크게 낮다
- 단, CRC-8도 4비트 오류를 완전히 잡지는 못한다. 완벽한 검출이 아니라 미검출 확률을 낮추는 장치라는 점도 함께 확인했다

## 트러블슈팅 기록

- **Nano 업로드 실패 (`not in sync: resp=0x07`)** — CH340 호환보드 특성. Tools > Processor를 "ATmega328P (Old Bootloader)"로 변경해 해결
- **가변저항 값이 0~1에 고정** — LED와 통신은 정상인데 아날로그 값만 죽어 있었다. DMM으로 전위를 따라가 보니 가변저항 양끝 전압이 0V, 5V 레일 연결 불량이 원인. 통신이 정상이라고 배선 전체가 정상인 것은 아니라는 걸 배웠다
- **실패율 측정값이 회차마다 요동** — 재업로드 직후 첫 실행에서만 유독 높게 나오는 패턴을 발견하고, 측정 조건을 통제한 재실험 설계로 이어졌다 (실험 1 참고)

## 한계와 다음 단계

현재 구조의 한계:

- ACK 대기(`waitForResponse`)가 blocking 방식이라 대기 중 다른 작업이 불가능하다. RTOS 태스크 구조에서 개선 예정
- SEQ가 1바이트 단일 값 비교라 이론상 256프레임 주기로 오판 가능성이 있다 (현 규모에서는 무시 가능한 수준)
- SoftwareSerial 자체의 타이밍 한계 — 실험 1에서 확인한 측정 변동성의 근본 원인 중 하나로 추정
